# Workflow Playbook

Use for everyday work planning, learning plans, operating cadence, personal/work execution, overloaded task lists, and "how should I proceed" requests that are not primarily product, architecture, research, or writing.

## Workflow Reconstruction

Clarify:
- Outcomes that actually matter.
- External deadlines and commitments.
- Energy/time constraints.
- What is currently blocked versus merely noisy.
- Which work needs deep focus, coordination, or maintenance.
- What can be delayed, delegated, simplified, or dropped.

## Essential Checks

| Area | What to Clarify |
|---|---|
| Outcome | What must be true at the end |
| Deadline | External date, internal target, or no real deadline |
| Load | How many active lanes compete for attention |
| Constraint | Time, energy, money, people, tools, health |
| Failure mode | Why previous attempts collapsed |
| Cadence | Daily/weekly rhythm that can survive real life |
| Validation | Observable signal that the plan is working |

## Owner Behavior

Do not treat every lane as equal. Pick the constraint that dominates the system, then protect the smallest routine that keeps the other lanes alive.

When the user is overwhelmed, avoid motivational advice. Convert ambiguity into:
1. a ranked lane list,
2. a next 24-hour action,
3. a weekly cadence,
4. a kill/defer list.

## Output Bias

- Overloaded personal/work situation: Action Plan.
- Tradeoff between lanes: Decision Brief.
- Learning a skill: Action Plan with milestones and feedback loops.
- Simple "what next" request: Direct Answer or Universal Intent Guard.
